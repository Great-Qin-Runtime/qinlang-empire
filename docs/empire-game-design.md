# 帝国游戏设计 / Empire Game Design

> 一郡一郡建国，一年一年扩疆，一代一代不绝。  
> 大秦不止于一统，永世以延帝国。

本文档是 **QinLang Empire 作为放置类经营游戏** 的核心设计。  
它取代了之前"语言博物馆"定位，是项目自此之后的**最高叙事文件**。

---

## 1. 一句话定义

> **大秦帝国是一个由 300+ 种编程语言协作构成的、永不结束的、放置式经营帝国。**  
> 玩家视角永远是 **秦**——从西陲一邑到一统六国，再到万世不绝。  
> 每一种语言都是帝国治下的一郡，承担一份真实的工作（产出 / 转换 / 服务 / 异术 / 庆典）。  
> 它自动运行：cron 心跳每 5 分钟一次，访客打开网页就能看到帝国在长。

## 2. 核心命题

| 命题 | 体现 |
|---|---|
| 用 300+ 种语言写一个项目 | 每种语言对应一个郡，提供其特长功能 |
| 不是 helloworld | 每郡输出 **真实改变帝国状态** 的代码 |
| 互相配合 | 帝国状态共享，资源在郡之间流动 |
| 自动运行 | tick 节拍 + GitHub Actions cron + 静态网页观看 |
| 永远是秦 | 玩家视角不切换；帝国从弱到强、一统、再延续 |
| 一统不结束 | 一统只是中段里程碑；之后进入帝国期与万世期 |

## 3. 时间与节拍

### 3.1 物理时间 ↔ 帝国时间

- **1 tick = 5 分钟**（GitHub Actions cron 默认）
- **1 帝国年 = 24 ticks = 2 小时**真实时间
- **1 朝代 ≈ 100~300 帝国年 = 8~25 天** 真实时间
- **从开服到一统**：约 **500 帝国年 ≈ 42 天 ≈ 6 周** 真实时间
- **一统之后**：永续运行，每天数十次 tick，每年数百事件

> 数字均为可调常量，V0.2 阶段定义在 `court/stages.py` 与 `.github/workflows/empire-tick.yml` 的 cron 表达式中。

### 3.2 后期节拍升级（路径 D）

第二阶段后引入：

- **服务器慢 tick**：5 分钟一次，写 `empire/state.json`，提交 git；
- **浏览器快 tick**：访客打开页面，本地以 1 秒一次的频率插值 / 模拟 / 演出动画；
- **快 tick 不改 server 状态**——只做视觉过渡，避免抢权威。

## 4. 朝代阶段（永远是秦的成长史）

```
秦邑 → 春秋 → 战国 → 横扫 → 一统 → 帝国 → 万世
   ↑                                              ↑
 起点                                          永不结束
```

| 阶段 ID | 名称 | 大约帝国年 | 真实时间 | 主题 | 解锁郡数 |
|---|---|---|---|---|---|
| `qin-yi` | 秦邑 | 0 ~ 100 | 4 天 | 立稳脚跟、积累基础 | 5 → 15 |
| `chun-qiu` | 春秋 | 100 ~ 300 | 8 天 | 与诸侯并立、边境贸易 | 15 → 50 |
| `zhan-guo` | 战国 | 300 ~ 450 | 6 天 | 富国强兵、改制变法 | 50 → 100 |
| `heng-sao` | 横扫 | 450 ~ 500 | 2 天 | 一年灭一国（六国轮灭） | 100 → 130 |
| `yi-tong` | 一统 | 500 | **片刻** | **车同轨、书同文、统一度量衡**——里程碑大事件 | 130 → 150 |
| `di-guo` | 帝国 | 500 ~ 800 | 12 天 | 巡狩、长城、驰道、设郡远征 | 150 → 250 |
| `wan-shi` | 万世 | 800 → ∞ | 永远 | 招贤纳士、远疆奇技、万邦来朝 | 250 → 300+ → 无上限 |

### 4.1 阶段晋升条件

每阶段晋升需要同时满足：

1. **资源阈值**（如：钱粮 ≥ X、户籍 ≥ Y、兵器 ≥ Z）
2. **里程碑事件**（如：商鞅变法、长平之战、灭韩、灭楚……）
3. **时间下限**（避免一夜跳代——例如至少经历 100 帝国年）

### 4.2 一统的特殊性

一统**不是**结束，而是**最盛大的一次仪式 tick**：

- 帝国年定格在 500，触发一次"超长 tick"；
- 一次性发动 30+ 郡协作（车同轨、书同文、度同制、币同形）；
- 一次性把帝国所有储备消耗一半，换取永久 buff；
- 之后进入 `di-guo` 阶段，**资源生产倍率全面提升**；
- 玉玺生成第一份"始皇玉玺"——所有访客可看到。

### 4.3 万世的设计

`wan-shi` 阶段刻意永不结束：

- 没有终结剧本，只有持续小事件（远征、奇灾、招贤、外族来朝、瘟疫、改制……）；
- 每解锁一个新郡 = 一次"招贤"事件，弹幕飘过郡名 + 来历；
- 资源不再是数字游戏，而变为 **多条文明指数**（武功 / 文治 / 工巧 / 学问 / 礼仪），各项独立增长，永无封顶；
- 每隔 N 帝国年触发一次"巡狩"——御史台抽查若干郡，跑全量协议自检，留档史册。

## 6. tick 流程（朝廷的一次心跳）

```
1. 调度器 (court/emperor.py) 加载 empire/state.json
2. 计算当前 tick 应运行的郡：
   - producer 类按各自 cooldown
   - service 类按事件触发
   - ceremonial 类按低概率随机
   - 每 tick 总共调度 5 ~ 30 个郡
3. 给每郡组装 dispatch（差遣）：
   - 当前帝国时间 / 阶段 / 季节 / 天气
   - 该郡相关的状态片段
   - 差遣类型（produce / transform / service / special / ceremony）
   - 随机种子（reproducible）
4. 子进程方式执行该郡 main.<ext>，stdin=dispatch JSON
5. 收集子进程 stdout：
   - deltas: 对状态的修改（资源 / 自身 / 邻郡）
   - events: 加入事件流
   - stamps: 仪式性盖章（保留以兼容老仪式）
6. 校验 schema → 合并 deltas → 推 events → 写回 state.json
7. tick 数 +1，必要时 year +1，触发阶段晋升判定
8. 写一行 history.jsonl（用于 dashboard 动画回放）
9. git commit + push（cron 模式）
```

## 7. 事件系统

```json
{
  "tick": 1234,
  "year": 56,
  "type": "produce | transform | service | military | natural | ceremonial | milestone | unlock",
  "from_province": "python",
  "text": "白蛇郡奉诏，献文书 5 卷",
  "severity": "info | warn | epic"
}
```

事件分级：

- `info`：普通生产、转换；dashboard 滚动条灰字；
- `warn`：天灾 / 外敌 / 资源不足；dashboard 黄字；
- `epic`：里程碑 / 阶段晋升 / 一统；dashboard 红字 + 全屏特效。

## 8. 解锁机制（招贤）

新郡（新语言）的加入对应游戏内的"招贤"事件：

- 一旦贡献者 PR 通过、新 manifest 入库；
- 下一个 tick 触发 `招贤` 事件：

```
帝国 87 年春，齐人 X 仕秦，献「<语言名>」之术。
帝国从此设「<郡名>」，列于<分类>之列。
```

- 该郡在 dashboard 地图上以"新地"高亮 N tick；
- 不需要其他玩家行为，**贡献本身即游戏内事件**——这就是"自动国家机器 = 放置游戏"的关键合一。

## 9. 失败与降级

任何郡若在某次 tick 中失败：

- `failed`：标记 `loyalty -5`；事件流写"X 郡奉诏不力"；
- `timeout`：标记 `loyalty -3`；事件流写"X 郡延误"；
- `protocol-violation`：标记 `loyalty -10`；连续 3 次触发"叛乱"事件；
- `setup-error`：不计 loyalty；事件流静默；
- 连续叛乱 → 郡被标记 `quarantined`，停止派遣，需 PR 修复。

> 这把 CI 失败 / schema 错误也游戏化了，不再是"红字警告"而是叙事的一部分。

## 10. Dashboard 视图

最简版本三块（M2 阶段）：

1. **庙堂** — 当前阶段 / 帝国年 / 资源储备 / 玉玺；
2. **舆图** — 所有郡的网格，按角色配色，活跃高亮；
3. **史册** — 事件流（最近 50 条），epic 事件钉顶。

完整版本（M5 阶段）：

- **天象** — 季节 / 天气 / 异象动画（GLSL 郡产出）；
- **百官** — 各类角色统计；
- **史馆** — 历代玉玺 + 重大里程碑回放；
- **远疆** — 万世期未来扩展占位。

## 11. 与原 Qin Law 协议的关系

旧协议：诏书是空文本 + 空 payload，stamps 是装饰。  
**新协议**：诏书 = **状态片段 + 差遣**；输出 = **deltas + events + stamps**。

stamps 字段保留——它从"装饰盖章"变成"史册记录的细粒度备份"，仍是事件流的一部分。  
旧的 `parade` / `chain` / `graph` 三种模式被 **`tick` 单一模式** 取代。

## 12. 不收编的语言（特区与外邦）

部分语言不适合进入主帝国 tick 系统：

- **HDL**（Verilog 等）：设"工部机关院"特区，独立离线 tick；产出"机关图"作为帝国奇观；
- **Proof**（Lean / Coq）：设"太学"特区，每帝国年一次提交"凭证"，加 `xue-wen` 资源；
- **Manual / Platform-Locked**（ABAP / Apex）：作为"外邦使节"，每月触发一次"朝贡"事件，产出固定资源；
- **Esolang**：进 `ceremonial` 角色，纯叙事彩蛋。

## 13. 永远是秦的视角实现

- `empire/state.json` 的根字段 `state.dynasty` 永远是 `"秦"`，不允许写入其他值；
- 没有"换文明"按钮；
- 一统后地图扩张但徽记不变；
- 万世期出现的"远疆奇术"以"献入大秦"叙事被吸收，**不会改变帝国主体**。

## 14. 玩家可干预度（很低，但存在）

- **无登录**：访客只是观看；
- **GitHub PR 即操作**：贡献新郡 / 修 bug = 在游戏内"献策"；
- **Issue 即 上书**：开 issue 会被 dashboard 渲染成"奏章一封"；
- **Star 即 颂圣**：star 数计入帝国"声望"指数（万世期 yi-li 加成）。

> 没有传统玩家 UI——**整个 GitHub 仓库就是游戏 UI**。

## 15. MVP 范围（M1）

**必须有的**：

- `empire/state.json` 初始状态（秦邑期，5 郡）；
- `court/emperor.py` 单 tick 实现；
- 5 个 MVP 郡（Python / C / SQL / Bash / Brainfuck）各自能产出真实 delta；
- `protocol/state.schema.json` + 更新后的 input/output/manifest schema；
- 极简 `dashboard/index.html`（读 state.json + history.jsonl）。

**可以没有的**（留给后续）：

- 阶段晋升逻辑（先固定在 qin-yi）；
- 完整事件系统（先只写 produce 类）；
- 招贤事件（M2 再做）；
- 浏览器快 tick；
- 文明指数 / 二阶资源 / 特区。
